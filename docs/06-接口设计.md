# 接口设计

## 1. 设计目标

接口设计用于约束 CampusOps 前后端交互方式，支撑后端 Controller、Service 和前端页面开发。

第一版接口以 REST 风格为主，优先覆盖用户认证、工单管理、知识库、AI 辅助、统计和日志查询。

## 2. 通用约定

### 2.1 URL 前缀

统一使用项目名前缀：

```text
/campusops
```

接口路径直接以项目名作为统一前缀，后续路径按业务模块划分。例如认证相关接口使用 `/campusops/auth/login`，工单相关接口使用 `/campusops/tickets`。

### 2.2 通用响应格式

```json
{
  "code": "SUCCESS",
  "message": "OK",
  "data": {},
  "traceId": "optional-trace-id"
}
```

错误响应示例：

```json
{
  "code": "TICKET_NOT_FOUND",
  "message": "工单不存在或无权限访问",
  "data": null,
  "traceId": "optional-trace-id"
}
```

### 2.3 分页请求

分页查询统一使用：

```text
page=1&pageSize=20
```

分页响应建议：

```json
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 100
}
```

### 2.4 鉴权方式

第一版使用 JWT Bearer Token：

```text
Authorization: Bearer <token>
```

前端登录成功后保存 Token，后续需要鉴权的请求统一携带 `Authorization` 请求头。

## 3. 认证与当前用户接口

### 3.1 获取图形验证码

```text
GET /campusops/auth/captcha
```

响应：

```json
{
  "captchaId": "uuid",
  "imageBase64": "data:image/png;base64,...",
  "expiresIn": 120
}
```

规则：

- 验证码为后端生成的一次性图形验证码。
- 验证码有效期 120 秒。
- 验证码校验成功或失败次数达到上限后失效。
- 第一版使用 JVM 内存存储验证码，不引入 Redis 或第三方验证码服务。

### 3.2 注册普通用户

```text
POST /campusops/auth/register
```

请求：

```json
{
  "username": "student002",
  "phone": "13800000007",
  "email": "student002@campusops.local",
  "password": "123456",
  "confirmPassword": "123456",
  "realName": "王同学",
  "department": "计算机学院",
  "captchaId": "uuid",
  "captchaCode": "A7K2"
}
```

响应：

```json
{
  "id": 7,
  "username": "student002",
  "realName": "王同学"
}
```

规则：

- `username` 必填，长度 4-64，只允许字母、数字、下划线。
- `phone` 必填，长度不超过 32，且必须符合手机号格式。
- `email` 必填，必须符合邮箱格式。
- `password` 必填，长度 6-32。
- `confirmPassword` 必填，必须与 `password` 一致。
- `realName` 必填，长度 1-64。
- `department` 可选，长度不超过 128。
- 注册前必须校验验证码；验证码错误不写入用户数据。
- 用户名、邮箱、手机号不能与未删除用户重复。
- 注册用户默认 `status = enabled`，默认绑定 `normal_user` 角色。
- 注册成功后不自动登录，由前端引导用户登录。

### 3.3 登录

```text
POST /campusops/auth/login
```

请求：

```json
{
  "account": "student001",
  "password": "123456",
  "captchaId": "uuid",
  "captchaCode": "A7K2"
}
```

响应：

```json
{
  "accessToken": "token",
  "tokenType": "Bearer",
  "expiresIn": 7200
}
```

规则：

- `account` 支持用户名、手机号或邮箱。
- 登录前必须校验验证码；验证码错误不进入密码校验。
- 为降低改动风险，后端可临时兼容旧字段 `username`，前端新实现必须使用 `account`。

### 3.4 获取当前用户信息

```text
GET /campusops/auth/me
```

响应：

```json
{
  "id": 1,
  "username": "student001",
  "realName": "张三",
  "email": "student001@campusops.local",
  "phone": "13800000001",
  "department": "计算机学院",
  "roles": ["normal_user"]
}
```

### 3.5 退出登录

```text
POST /campusops/auth/logout
```

第一版如果使用无状态 JWT，可以由前端清除 Token，后端接口可选。

## 4. 工单接口

### 4.1 创建工单

```text
POST /campusops/tickets
```

普通用户创建工单。

请求：

```json
{
  "title": "校园网无法连接",
  "description": "宿舍网络一直断开，认证页面无法打开。",
  "categoryId": 1,
  "priority": "medium"
}
```

响应：

```json
{
  "id": 1001,
  "ticketNo": "INC-2026-0001",
  "status": "pending_assignment"
}
```

### 4.2 查询工单列表

```text
GET /campusops/tickets
```

查询参数：

- `status`
- `categoryId`
- `priority`
- `submitterId`
- `assigneeId`
- `keyword`
- `page`
- `pageSize`

权限规则：

- 普通用户只能查询自己提交的工单。
- IT 处理人只能查询自己提交的工单、分配给自己的工单和授权范围内的工单。
- 服务管理员可以查询全部工单。

### 4.3 查看工单详情

```text
GET /campusops/tickets/{ticketId}
```

返回内容应包含：

- 工单基础信息。
- 分类、优先级、状态。
- 提交人和处理人信息。
- 处理记录。
- 评论。
- 附件。
- AI 摘要和推荐结果。

### 4.4 更新工单基础信息

```text
PUT /campusops/tickets/{ticketId}
```

用于修改标题、描述、分类、优先级等基础信息。

权限限制：

- 普通用户只能在工单未被处理前修改自己的标题和描述。
- 服务管理员可以调整分类和优先级。

### 4.5 分派工单

```text
POST /campusops/tickets/{ticketId}/assign
```

请求：

```json
{
  "assigneeId": 20,
  "reason": "分派给网络运维处理"
}
```

权限：服务管理员。

状态变化：`pending_assignment` -> `pending_process`。

### 4.6 接单

```text
POST /campusops/tickets/{ticketId}/accept
```

权限：被分派的 IT 处理人。

状态变化：`pending_process` -> `processing`。

### 4.7 添加处理记录

```text
POST /campusops/tickets/{ticketId}/records
```

请求：

```json
{
  "content": "已联系用户确认网络环境，准备现场排查。",
  "visibleToUser": true
}
```

权限：当前处理人或服务管理员。

### 4.8 添加评论

```text
POST /campusops/tickets/{ticketId}/comments
```

请求：

```json
{
  "content": "我补充了一张错误截图。"
}
```

权限：工单提交人、当前处理人、服务管理员。

### 4.9 标记已解决

```text
POST /campusops/tickets/{ticketId}/resolve
```

请求：

```json
{
  "solution": "重置账号绑定状态后恢复正常。"
}
```

权限：当前处理人或服务管理员。

状态变化：`processing` -> `pending_confirm`。

### 4.10 用户确认关闭

```text
POST /campusops/tickets/{ticketId}/close
```

请求：

```json
{
  "score": 5,
  "comment": "处理及时，问题已解决。"
}
```

权限：工单提交人。

状态变化：`pending_confirm` -> `closed`。

### 4.11 取消工单

```text
POST /campusops/tickets/{ticketId}/cancel
```

权限：工单提交人。

限制：工单未进入处理中前允许取消。

### 4.12 驳回工单

```text
POST /campusops/tickets/{ticketId}/reject
```

请求：

```json
{
  "reason": "该问题不属于校园 IT 服务范围。"
}
```

权限：服务管理员。

## 6. 知识库接口

### 6.1 查询知识库文章

```text
GET /campusops/knowledge/articles
```

查询参数：

- `keyword`
- `categoryId`
- `status`
- `page`
- `pageSize`

普通用户默认只能查看已发布文章。

### 6.2 查看文章详情

```text
GET /campusops/knowledge/articles/{articleId}
```

### 6.3 创建文章草稿

```text
POST /campusops/knowledge/articles
```

权限：IT 处理人、服务管理员。

### 6.4 更新文章

```text
PUT /campusops/knowledge/articles/{articleId}
```

权限：作者、服务管理员。

### 6.5 发布文章

```text
POST /campusops/knowledge/articles/{articleId}/publish
```

权限：服务管理员。

### 6.6 从工单生成知识库草稿

```text
POST /campusops/knowledge/articles/from-ticket/{ticketId}
```

权限：IT 处理人、服务管理员。

说明：可调用 AI 生成草稿，但发布仍需人工确认。

## 7. AI 辅助接口

AI 接口只生成建议，不直接修改正式业务状态。

### 7.1 分析工单

```text
POST /campusops/ai/tickets/{ticketId}/analyze
```

功能：生成分类建议、摘要和关键词。

权限：有权查看该工单的用户。

### 7.2 获取工单 AI 分析结果

```text
GET /campusops/ai/tickets/{ticketId}/analysis
```

### 7.3 推荐相似工单

```text
POST /campusops/ai/tickets/{ticketId}/similar-tickets
```

权限：有权查看该工单，并且推荐结果也必须经过数据权限过滤。

### 7.4 推荐知识库文章

```text
POST /campusops/ai/tickets/{ticketId}/knowledge-recommendations
```

### 7.5 生成知识库草稿

```text
POST /campusops/ai/tickets/{ticketId}/knowledge-draft
```

权限：IT 处理人、服务管理员。

说明：只生成草稿内容，不直接发布文章。

### 7.6 查询 AI 调用记录

```text
GET /campusops/ai/analysis-records
```

权限：系统管理员、服务管理员。

## 8. 统计接口

### 8.1 工单概览统计

```text
GET /campusops/statistics/tickets/overview
```

返回：

- 工单总数。
- 待分派数量。
- 处理中数量。
- 已关闭数量。
- 超时数量。

### 8.2 分类统计

```text
GET /campusops/statistics/tickets/by-category
```

### 8.3 处理人工作量统计

```text
GET /campusops/statistics/tickets/by-assignee
```

### 8.4 处理时长统计

```text
GET /campusops/statistics/tickets/processing-time
```

权限：服务管理员。

### 9.3 操作日志查询

```text
GET /campusops/admin/operation-logs
```

权限：系统管理员。

## 10. 附件接口

### 10.1 上传工单附件

```text
POST /campusops/tickets/{ticketId}/attachments
```

Content-Type：`multipart/form-data`

权限：工单提交人、当前处理人、服务管理员。

### 10.2 下载附件

```text
GET /campusops/attachments/{attachmentId}/download
```

权限：必须有权访问附件所属工单。
