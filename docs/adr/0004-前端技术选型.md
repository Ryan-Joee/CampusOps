# ADR 0004：前端技术选型

## 状态

已接受

## 背景

CampusOps 前端需要支撑校园 IT 工单工作台，包括登录、工单列表、工单详情、知识库、统计分析、系统管理和 AI 辅助面板等页面。

项目目标以 Java 后端业务系统和 AI Agent 辅助开发实践为主，前端需要保证清晰、可用、易维护，但不应成为第一阶段的主要复杂度来源。

开发者目前更熟悉 JavaScript，尚未系统学习 TypeScript。

## 决策

前端第一阶段采用以下技术栈：

- Vue 3
- Vite
- JavaScript
- Vue Router
- Pinia
- Element Plus
- Axios
- ECharts

第一阶段不使用 TypeScript。

## 原因

选择 Vue 3 和 Vite 的原因：

- Vue 3 适合构建管理后台和业务工作台。
- Vite 启动快、配置简单，适合毕业设计快速开发。
- Vue 生态成熟，资料丰富，便于 AI Agent 辅助开发。

选择 JavaScript 而不是 TypeScript 的原因：

- 降低前端学习成本。
- 避免 TypeScript 学习压力影响项目主线。
- 第一阶段页面复杂度可控，使用 JavaScript 足以完成业务闭环。
- 后续如果项目继续维护，可以再评估迁移 TypeScript。

选择 Element Plus 的原因：

- 适合管理后台和工作台类产品。
- 提供表格、表单、分页、菜单、弹窗、标签、日期选择等常用组件。
- 可以减少前端基础组件开发成本。

选择 Pinia、Vue Router、Axios 和 ECharts 的原因：

- Vue Router 用于页面路由。
- Pinia 用于管理当前用户、Token、角色和菜单状态。
- Axios 用于封装后端 API 请求。
- ECharts 用于工单统计图表。

## 前端产品形态

CampusOps 前端应采用管理后台和工单工作台形态。

优先实现：

- 登录页。
- 首页仪表盘。
- 我的工单。
- 待处理工单。
- 工单详情页。
- 知识库页面。
- 服务分类管理。
- 用户与角色管理。
- 统计分析页面。
- AI 辅助面板。

第一阶段不做：

- 营销首页。
- 炫酷大屏。
- 复杂动画。
- 低代码页面设计器。
- 微前端。
- SSR / Nuxt。

## 前端目录建议

建议目录结构：

```text
campusops-front/
  src/
    api/
    assets/
    components/
    constants/
    layouts/
    router/
    stores/
    utils/
    views/
      auth/
      dashboard/
      ticket/
      knowledge/
      admin/
      statistics/
```

## 后续演进

如果项目毕业后继续维护，并且前端规模明显扩大，可以重新评估：

- 是否迁移 TypeScript。
- 是否增加更严格的 ESLint / Prettier 规则。
- 是否拆分更细的组件和业务模块。
- 是否引入更完善的前端测试。

这些决策需要通过新的 ADR 记录。
