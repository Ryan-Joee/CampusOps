# ADR 0003：数据库初始化策略

## 状态

已接受

## 背景

CampusOps 第一阶段需要创建用户、角色、工单、知识库、AI 分析记录和日志等数据库表。

项目当前处于本科毕业设计启动阶段，开发者尚未学习 Flyway、Liquibase 等数据库迁移工具。

为了降低学习成本和项目复杂度，需要选择简单、直观、可控的数据库初始化方式。

## 决策

第一阶段不引入 Flyway 或 Liquibase。

数据库初始化采用普通 SQL 文件方式：

```text
campusops-server/src/main/resources/sql/schema.sql
campusops-server/src/main/resources/sql/data.sql
```

其中：

- `schema.sql` 负责创建数据库表结构。
- `data.sql` 负责初始化基础数据和演示数据。

## 原因

选择普通 SQL 文件的原因：

- 学习成本低。
- 便于直接理解数据库结构。
- 适合毕业设计第一阶段。
- 方便在 MySQL 客户端或 IDEA 中手动执行。
- 不需要额外引入数据库迁移工具依赖。

暂不选择 Flyway 或 Liquibase 的原因：

- 当前项目还处于启动阶段。
- 开发者之前没有使用经验。
- 第一版表结构变化可通过文档和 SQL 文件维护。
- 过早引入会增加认知负担。

## SQL 文件职责

### schema.sql

应包含：

- 用户与权限表。
- 工单核心表。
- 知识库表。
- AI 分析记录表。
- 操作日志表。
- 必要索引。

### data.sql

应包含：

- 初始角色。
- 初始管理员用户。
- 初始工单分类。
- 初始知识库分类。
- 少量示例用户。
- 少量示例工单。

## 后续演进

当项目进入长期维护阶段，且数据库变更频率增加时，可以重新评估是否引入 Flyway 或 Liquibase。

如果未来引入数据库迁移工具，需要新增 ADR 说明原因、迁移方式和兼容策略。
