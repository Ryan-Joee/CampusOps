# 14-后端分层与包结构规范

## 目标

本文档约束 `campusops-server` 的 Java 包结构和分层职责，后续所有后端任务都必须遵守。AI Agent 编码前应先阅读本文档，不允许自行发明新的分层风格。

## 总体原则

- 本项目采用传统 MVC 技术分层单体架构，**不采用按业务域作为一级包的模块化单体结构**。
- 一级包按技术层划分：`controller`、`service`、`mapper`、`entity`、`dto`、`vo`、`security`、`config`、`common`。
- 不提前为微服务拆分做目录设计。当前阶段所有代码在同一个 Spring Boot 应用中，按技术职责分层即可。
- Controller、Service、Mapper、Entity、DTO、VO 职责必须清晰分离。
- 通用能力放在 `common`、`config`、`security` 下。
- 不允许 Controller 直接访问 Mapper。
- 不允许 Entity 直接作为接口请求或响应对象暴露给前端。
- 不允许把多个业务域的逻辑混在一个 Service 或 Controller 类中（通过方法命名和类命名区分业务域，而不是通过一级包）。

## 推荐包结构

```text
com.campusops
  controller                   # 控制器，按业务域命名类
    AuthController
    (future) TicketController
    (future) KnowledgeController

  service                      # 服务接口
    AuthService
    UserQueryService
    (future) TicketService
    (future) KnowledgeService

  service.impl                 # 服务实现
    AuthServiceImpl
    UserQueryServiceImpl
    (future) TicketServiceImpl

  mapper                       # MyBatis Mapper 接口
    SysUserMapper
    SysRoleMapper
    (future) SysTicketMapper

  entity                       # 数据库实体
    SysUserEntity
    SysRoleEntity
    (future) SysTicketEntity

  dto                          # 请求入参 DTO，按业务域分子包
    auth
      LoginRequest
    ticket
    knowledge
    ai

  vo                           # 响应出参 VO，按业务域分子包
    auth
      LoginResponse
      CurrentUserVO
    ticket
    knowledge
    ai

  security
    config                     # Spring Security 配置
    jwt                        # JWT 生成、解析、校验
    filter                     # 认证过滤器
    handler                    # 认证失败、无权限处理

  config
    mybatis                    # MyBatis Plus 配置
    openapi                    # Swagger / OpenAPI 配置
    web                        # Web MVC、跨域等配置（按需）

  common
    result                     # 统一响应、错误码
    exception                  # 业务异常、全局异常处理
    page                       # 分页对象
    constant                   # 通用常量
    util                       # 通用工具类，谨慎新增
```

## 分层职责

### Controller

- 只处理 HTTP 入参、参数校验、当前用户上下文和响应包装。
- 调用 Service，不写业务规则，不直接调用 Mapper。
- 入参使用 `dto` 下对应业务域子包中的类，出参使用 `vo` 下对应业务域子包中的类。

### Service

- 承载业务规则、事务边界、权限相关业务判断和跨表协作。
- 接口放在 `service` 包，实现放在 `service.impl` 包。
- 可调用 Mapper，也可通过其他 Service 完成跨业务协作。

### Mapper

- 只负责数据库访问。
- Mapper 接口放在 `mapper` 包，XML 放在 `resources/mapper/` 下。
- 不写业务判断，不拼接接口响应对象。

### Entity

- 与数据库表结构对应。
- 字段命名使用 Java 驼峰，数据库字段使用下划线。
- 不作为 Controller 的请求或响应对象。

### DTO / VO

- `dto` 按业务域分子包（如 `dto.auth`、`dto.ticket`），存放请求入参，例如 `LoginRequest`、`TicketCreateRequest`。
- `vo` 按业务域分子包（如 `vo.auth`、`vo.ticket`），存放响应出参，例如 `LoginResponse`、`TicketDetailVO`。
- 不把数据库实现细节泄漏给前端。

### Convert

- 当对象转换逻辑变多时，可在 `common` 下新增 `convert` 包，或按业务域命名转换类放在对应的 `dto`/`vo` 子包附近。
- 简单转换可以先写在 Service 内，避免过早抽象。

### Logging

- 后端统一使用 SLF4J 日志门面。当前项目已通过 Spring Boot 默认日志体系和 Lombok 支持日志能力，正常情况下不新增日志依赖。
- 需要记录日志的类优先使用 Lombok `@Slf4j`，不要手写 `LoggerFactory`，除非有明确原因。
- 日志规范适用于整个后端项目，不只适用于某个任务。所有新增或修改后端业务代码时，都要检查是否需要补充日志。
- Service 层应在关键业务节点记录 `log.info`，例如注册成功、登录成功、工单创建、工单查询权限命中、状态流转、分类或知识库维护等。
- 业务失败、权限拒绝、验证码错误、重复提交、数据不存在或状态不允许等可预期问题使用 `log.warn`。
- 未预期系统异常使用 `log.error`，并保留异常堆栈；全局异常处理器统一兜底。
- 日志不得输出明文密码、Token、验证码答案、完整身份证明材料等敏感信息。
- 日志字段优先记录业务 ID、用户 ID、工单编号、状态、错误码等可定位信息，避免输出完整请求体。
- Controller 层不堆业务日志，除非是请求入口级别的审计信息；业务过程日志主要放在 Service 层。
- Mapper、Entity、DTO、VO 一般不记录日志。
- 新增业务功能的验收应包含日志检查：关键成功路径、关键拒绝路径和异常路径是否有适度日志，且没有敏感信息泄漏。

## 命名约定

- Controller：`XxxController`
- Service 接口：`XxxService`
- Service 实现：`XxxServiceImpl`
- Mapper：`XxxMapper`
- Entity：`XxxEntity`
- 请求 DTO：`XxxRequest`
- 响应 VO：`XxxVO` 或 `XxxResponse`
- 枚举：`XxxStatusEnum`、`XxxTypeEnum`

## 与旧版结构的关键差异

| 项目 | 旧版（按业务域分包） | 新版（MVC 技术分层） |
|------|---------------------|---------------------|
| 一级包 | auth, user, ticket, knowledge, ai | controller, service, mapper, entity, dto, vo |
| Controller 位置 | auth.controller, ticket.controller | controller 下统一管理 |
| Service 位置 | auth.service, user.service | service / service.impl 下统一管理 |
| Mapper 位置 | user.mapper | mapper 下统一管理 |
| Entity 位置 | user.entity | entity 下统一管理 |
| DTO/VO | 各业务模块下的 dto/vo | dto / vo 下按业务域分子包 |

## AI Agent 编码约束

- 每次任务只允许修改任务文档中声明的范围。
- 新增业务代码前，先确认类的技术分层归属（controller / service / mapper / entity / dto / vo）。
- 不允许为了快速实现把类放到根包、临时包或 `misc`、`helper`、`manager` 等含义模糊的包。
- 不允许提交 `.agents/`、`.codex/`、`.claude/` 等本地 Agent 工具目录。
- 如需引入新依赖，必须先说明用途、版本兼容性和替代方案。

## 当前阶段说明

当前项目处于 MVC 分层单体架构阶段，所有代码按技术层组织。后续如有需要拆分为微服务，应通过 ADR 文档明确决策后再调整包结构。
