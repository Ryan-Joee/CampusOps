# 14-后端分层与包结构规范

## 目标

本文档约束 `campusops-server` 的 Java 包结构和分层职责，后续所有后端任务都必须遵守。AI Agent 编码前应先阅读本文档，不允许自行发明新的分层风格。

## 总体原则

- 采用按业务域划分模块的单体架构，不采用 JPA Repository 风格。
- Controller、Service、Mapper、Entity、DTO、VO 职责必须清晰分离。
- 通用能力放在 `common`、`config`、`security` 下，业务能力放在各业务模块下。
- 不允许 Controller 直接访问 Mapper。
- 不允许 Entity 直接作为接口请求或响应对象暴露给前端。
- 不允许把多个业务域的逻辑混在一个 `service` 或 `controller` 包中。

## 推荐包结构

```text
com.campusops
  common
    result          # 统一响应、错误码

    util            # 通用工具类，谨慎新增

    mybatis         # MyBatis Plus 配置

    jwt             # JWT 生成、解析、校验

    service
    service.impl
    dto
    vo

  user
    controller
    service
    service.impl
    mapper
    entity
    dto

  ticket
    controller
    service
    service.impl
    mapper
```

## 分层职责

### Controller

- 只处理 HTTP 入参、参数校验、当前用户上下文和响应包装。
- 调用 Service，不写业务规则，不直接调用 Mapper。
- 入参使用 `dto`，出参使用 `vo`。

### Service

- 承载业务规则、事务边界、权限相关业务判断和跨表协作。
- 接口放在 `service`，实现放在 `service.impl`。
- 可调用本模块 Mapper，也可通过其他模块 Service 完成跨模块协作。

### Mapper

- 只负责数据库访问。
- Mapper 接口放在各模块 `mapper` 包，XML 放在 `resources/mapper/模块名/` 下。
- 不写业务判断，不拼接接口响应对象。

### Entity

- 与数据库表结构对应。
- 字段命名使用 Java 驼峰，数据库字段使用下划线。
- 不作为 Controller 的请求或响应对象。

### DTO / VO

- `dto` 表示请求入参，例如 `LoginRequest`、`TicketCreateRequest`。
- `vo` 表示响应出参，例如 `LoginResponse`、`TicketDetailVO`。
- 不把数据库实现细节泄漏给前端。

### Convert

- 当对象转换逻辑变多时，可在模块内新增 `convert` 包。
- 简单转换可以先写在 Service 内，避免过早抽象。

## 命名约定

- Controller：`XxxController`
- Service 接口：`XxxService`
- Service 实现：`XxxServiceImpl`
- Mapper：`XxxMapper`
- Entity：`XxxEntity`
- 请求 DTO：`XxxRequest`
- 响应 VO：`XxxVO` 或 `XxxResponse`
- 枚举：`XxxStatusEnum`、`XxxTypeEnum`

## AI Agent 编码约束

- 每次任务只允许修改任务文档中声明的范围。
- 新增业务代码前，先确认应该归属哪个业务模块。
- 不允许为了快速实现把类放到根包、临时包或 `misc`、`helper`、`manager` 等含义模糊的包。
- 不允许提交 `.agents/`、`.codex/`、`.claude/` 等本地 Agent 工具目录。
- 如需引入新依赖，必须先说明用途、版本兼容性和替代方案。

## 当前阶段说明

当前项目仍处于基础骨架阶段，允许部分业务模块只有 `package-info.java`。从登录认证任务开始，新增代码必须按本文档的模块结构落位。
