# AI 模块设计

## 1. 设计目标

AI 模块用于增强工单处理效率和知识复用能力，但不替代传统业务系统。

AI 模块应满足以下目标：

- 提高工单分类、摘要和推荐效率。
- 辅助处理人快速理解问题。
- 辅助管理员发现高频问题和服务风险。
- 支持未来演进到 RAG、工具调用、Agent 和 MCP。
- 保证 AI 能力具备权限边界、调用日志和失败降级。

## 2. AI 能做什么

第一版 AI 能力包括：

- 根据工单标题和描述生成分类建议。
- 根据工单标题、描述、评论和处理记录生成摘要。
- 提取关键词。
- 推荐相似历史工单。
- 推荐相关知识库文章。

后续可扩展能力：

- 生成用户回复草稿。
- 生成处理方案草稿。
- 生成知识库文章草稿。
- 生成周报和月报。
- 基于 RAG 的知识库问答。
- Agent 辅助诊断。
- AI 工具调用。

## 3. AI 不能做什么

AI 不允许直接执行以下操作：

- 创建正式工单。
- 删除工单。
- 关闭工单。
- 修改工单处理人。
- 修改工单优先级。
- 发布知识库文章。
- 修改用户权限。
- 删除审计日志。

AI 可以生成草稿或建议，但必须由具备权限的人类用户确认后，才可以通过后端业务接口执行。

## 4. 第一版 AI 能力设计

### 4.1 工单分类建议

输入：

- 工单标题
- 工单描述
- 用户选择的初始分类，可选

输出：

- 建议分类
- 置信度
- 关键词
- 简短理由

示例分类：

- 网络问题
- 账号权限
- 系统故障
- 软件环境
- 设备故障
- 咨询类问题

### 4.2 工单摘要

输入：

- 工单标题
- 工单描述
- 用户补充信息
- 处理记录

输出：

- 问题摘要
- 当前进展
- 待处理事项

### 4.3 相似工单推荐

第一版可以先使用关键词匹配或数据库检索实现。

后续引入向量检索，实现语义相似推荐。

输出：

- 相似工单 ID
- 相似工单标题
- 相似原因
- 历史解决状态

### 4.4 知识库推荐

第一版可以基于分类和关键词匹配。

后续可以结合向量检索和 RAG。

输出：

- 知识库文章 ID
- 标题
- 摘要
- 推荐理由

## 5. AI 调用记录

每次 AI 调用应记录：

- 调用场景
- 关联业务对象 ID
- 输入摘要
- 输出摘要
- 模型名称
- 调用耗时
- 调用状态
- 错误信息
- 创建时间

敏感字段不应完整明文记录到日志中。

## 6. 失败降级

AI 调用失败时，系统应保证核心业务流程不受影响。

降级策略：

- 分类建议失败时，允许管理员手动分类。
- 摘要失败时，直接展示原始工单内容。
- 推荐失败时，隐藏推荐区域或展示空结果。
- 模型服务不可用时，记录失败日志并提示稍后重试。

## 7. 权限约束

AI 模块只能处理当前用户有权限访问的数据。

例如：

- 普通用户请求 AI 总结时，只能总结自己的工单。
- IT 处理人请求相似工单时，只能检索其有权限访问的历史工单。
- 服务管理员可以在全部工单范围内进行统计和推荐。

## 8. 长期演进

### 8.1 RAG

对知识库文章和历史工单进行向量化，使 AI 能基于可追溯资料回答问题。

### 8.2 工具调用

AI 可以调用受控后端工具，例如：

- 查询工单列表。
- 查询知识库文章。
- 生成工单草稿。
- 生成知识库草稿。
- 生成运营报告。

工具调用结果必须经过权限校验。

### 8.3 Agent 工作流

后续可以拆分为：

- 分类 Agent
- 诊断 Agent
- 知识推荐 Agent
- 报告生成 Agent
- 知识维护 Agent

### 8.4 MCP

后续可以通过 MCP 接入外部系统，例如日志平台、监控系统、文档系统、通知系统等。
