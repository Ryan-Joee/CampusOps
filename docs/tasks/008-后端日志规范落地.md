# 008-后端日志规范落地

## 任务目标

在 `campusops-server/` 中统一落地后端日志规范，对当前已有关键业务路径补充适度日志，并为后续任务建立一致写法。

本任务不是只服务注册接口，而是面向整个后端项目。所有后续后端业务开发都必须遵守 `docs/14-后端分层与包结构规范.md` 的 Logging 章节。

## 开发前必读

请先阅读：

- `CLAUDE.md`
- `docs/14-后端分层与包结构规范.md`
- `docs/tasks/002-登录认证与当前用户接口.md`
- `docs/tasks/004-工单创建和查询.md`
- `docs/tasks/006-用户注册接口.md`

## 技术约束

- 后端统一使用 SLF4J 日志门面。
- 当前项目已有 Spring Boot 默认日志体系和 Lombok，不新增日志依赖。
- 优先使用 Lombok `@Slf4j`。
- 不手写 `LoggerFactory`，除非有明确理由。
- 不修改数据库表结构。
- 不修改前端代码。
- 不新增 `com.campusops.auth/user/ticket/knowledge/ai` 旧顶层业务包。

## 日志覆盖范围

至少检查并补充以下类：

- `campusops-server/src/main/java/com/campusops/service/impl/AuthServiceImpl.java`
- `campusops-server/src/main/java/com/campusops/service/impl/UserQueryServiceImpl.java`
- `campusops-server/src/main/java/com/campusops/service/impl/TicketServiceImpl.java`
- `campusops-server/src/main/java/com/campusops/common/exception/GlobalExceptionHandler.java`
- 如 006 已实现验证码和注册，还需检查：
  - `campusops-server/src/main/java/com/campusops/service/impl/CaptchaServiceImpl.java`
  - 注册相关 Service 实现代码

## 需要记录的关键日志

### 认证与用户

- 登录成功：`log.info`，记录 `userId`、`username`、角色数量或角色编码，不记录 Token。
- 登录失败：`log.warn`，记录失败原因类型，例如账号不存在、密码错误、账号禁用、验证码错误；不记录明文密码。
- 获取当前用户失败：`log.warn`，记录 `userId` 和错误码。
- 注册成功：`log.info`，记录 `userId`、`username`、是否填写手机号和邮箱。
- 注册失败：`log.warn`，记录失败原因，例如用户名重复、邮箱重复、手机号重复、验证码错误。

### 验证码

- 生成验证码：`log.info`，记录 `captchaId`、过期时间，不记录验证码答案。
- 校验失败：`log.warn`，记录 `captchaId` 和失败原因，不记录验证码答案。
- 验证码过期或不存在：`log.warn`。

### 工单

- 创建工单成功：`log.info`，记录 `ticketId`、`ticketNo`、`submitterId`。
- 查询工单列表：可记录 `log.info` 或 `log.debug`，记录当前用户 ID、角色类型、筛选条件、分页参数；不要输出完整查询结果。
- 查看工单详情成功：`log.info`，记录 `ticketId` 和当前用户 ID。
- 无权查看工单或工单不存在：`log.warn`，记录 `ticketId`、当前用户 ID 和错误码。
- 参数非法、状态非法、分类不存在：`log.warn`。

### 全局异常

- 业务异常保持 `log.warn`。
- 未捕获异常保持 `log.error` 并输出堆栈。
- 参数校验异常可以使用 `log.warn`，但不要输出完整请求体。

## 敏感信息禁止记录

日志中不得输出：

- 明文密码。
- JWT Token。
- 验证码答案。
- 完整请求体。
- 完整手机号和邮箱在不必要场景下的明文；如需记录，优先记录是否填写或脱敏后值。
- 其他可能识别个人身份的敏感信息。

## 建议实现方式

- 需要日志的 Service 实现类添加 `@Slf4j`。
- 使用结构化占位符写法：

```java
log.info("工单创建成功: ticketId={}, ticketNo={}, submitterId={}", ticketId, ticketNo, submitterId);
```

- 不使用字符串拼接：

```java
log.info("工单创建成功: " + ticketNo);
```

- 不在循环中输出高频日志，除非有明确排障价值。
- 不为了“看起来企业级”而到处加日志；只在关键业务边界、状态变化、权限拒绝和异常路径加。

## 验收标准

必须全部满足：

- 关键认证、注册、验证码、工单路径有适度日志。
- 日志使用 `@Slf4j` 和 `log.info/log.warn/log.error`。
- 没有新增日志依赖。
- 没有明文密码、Token、验证码答案泄漏。
- `mvn -q test` 通过。
- 旧顶层业务包检查无匹配。

## 验收命令

```powershell
git -c core.excludesfile= status --short
cd campusops-server
mvn -q test
cd ..
rg -n "password=.*|token=.*|captchaCode|验证码答案|Authorization" campusops-server/src/main/java
rg -n "^package com\.campusops\.(auth|user|ticket|knowledge|ai)(;|\.)" campusops-server/src/main/java
```

说明：

- 第一条 `rg` 用于辅助发现明显敏感日志风险，发现匹配时需要人工判断是否确实泄漏。
- 第二条 `rg` 应无匹配。

## 明确不要做

- 不引入 Log4j2、Logstash、ELK、Sentry 等新组件。
- 不新增 AOP 日志系统。
- 不做操作审计表写入。
- 不改前端。
- 不改数据库结构。
- 不把日志写到业务响应里。
