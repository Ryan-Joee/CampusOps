# 004-工单创建和查询

## 任务目标

在 `campusops-server/` 中实现工单基础闭环的第一步：当前登录用户创建工单、按权限分页查询工单列表、按权限查看工单详情。

本任务只覆盖工单主表 `ticket` 的创建和查询能力，为后续分派、接单、处理记录、评论、附件、关闭和 AI 分析打基础。

## 本次实现范围

### 1. 创建工单

```text
POST /campusops/tickets
```

Controller 实际映射：

```text
POST /tickets
```

当前登录用户创建工单。

请求字段：

- `title`：必填，长度 1-200。
- `description`：必填。
- `categoryId`：可选；如果传入，必须对应启用且未删除的分类。
- `priority`：可选，默认 `medium`；允许值：`low`、`medium`、`high`、`urgent`。

响应 `data`：

```json
{
  "id": 1001,
  "ticketNo": "INC-2026-0001",
  "status": "pending_assignment"
}
```

创建规则：

- `submitter_id` 使用当前登录用户 ID。
- `status` 固定为 `pending_assignment`。
- `source` 固定为 `web`。
- `priority` 未传时使用 `medium`。
- `ticket_no` 格式使用 `INC-yyyy-xxxx`，例如 `INC-2026-0004`，保证唯一即可，不引入分布式 ID 或新依赖。

### 2. 查询工单列表

```text
GET /campusops/tickets
```

Controller 实际映射：

```text
GET /tickets
```

查询参数：

- `status`：可选。
- `categoryId`：可选。
- `keyword`：可选，匹配 `ticket_no` 或 `title`。
- `page`：默认 1，最小 1。
- `pageSize`：默认 10，范围 1-100。

响应类型：

```text
ApiResponse<PageResult<TicketListItemVO>>
```

`TicketListItemVO` 至少包含：

- `id`
- `ticketNo`
- `title`
- `categoryId`
- `categoryName`
- `priority`
- `status`
- `submitterId`
- `submitterName`
- `assigneeId`
- `assigneeName`
- `createdAt`
- `updatedAt`

### 3. 查看工单详情

```text
GET /campusops/tickets/{ticketId}
```

Controller 实际映射：

```text
GET /tickets/{ticketId}
```

`TicketDetailVO` 至少包含：

- `id`
- `ticketNo`
- `title`
- `description`
- `categoryId`
- `categoryName`
- `priority`
- `status`
- `submitterId`
- `submitterName`

本任务详情接口不返回处理记录、评论、附件、AI 推荐结果，这些留给后续任务。

## 权限规则

数据权限必须在 Service 层显式处理，Controller 不直接访问 Mapper。

- `normal_user`：只能查看自己提交的工单，可以创建工单。
- `technician`：可以查看自己提交的工单，以及分配给自己的工单。
- `service_admin`：可以查看全部工单。
- `system_admin`：不默认绕过业务权限；如果没有同时拥有 `service_admin` 或 `technician`，只按普通提交人逻辑处理。

详情接口无权访问时，返回统一错误，不暴露工单是否存在。

## 状态和优先级

工单状态允许值：

- `pending_assignment`
- `pending_process`
- `processing`
- `pending_confirm`
- `closed`
- `rejected`
- `canceled`

优先级允许值：

- `low`
- `medium`
- `high`
- `urgent`

可以使用字符串常量或枚举实现，但不要为了本任务引入复杂状态机。

## Mapper 设计要求

`TicketMapper` 继承 `BaseMapper<TicketEntity>`。

XML 至少提供：

- `selectTicketPage`
- `countTicketPage`
- `selectTicketDetail`

查询要求：

- 必须过滤 `ticket.deleted = 0`。
- 列表和详情需要按当前用户角色附加数据权限条件。
- 需要 `LEFT JOIN ticket_category` 查询分类名称。
- 需要 `LEFT JOIN sys_user` 查询提交人和处理人姓名。
- Mapper 不做业务权限决策，只接收 Service 计算后的查询条件。

## 测试要求

- 当前用户创建工单成功，状态为 `pending_assignment`。

- `service_admin` 可以查看全部工单。

- 不出现旧顶层业务包 `com.campusops.ticket`。

优先补 Service 单元测试；必要时补 Controller WebMvcTest。不要为了测试引入 H2 或 Testcontainers。
