# 007-前端注册登录与主界面入口

## 任务目标

在 `campusops-front/` 中完成真实账号入口闭环：图形验证码、注册账号、登录、保存 Token、恢复当前用户、退出登录，并进入主界面。

本任务替代旧的 `docs/tasks/005-前端登录与当前用户接入.md`。旧任务中“只做登录、不做注册”“展示测试账号”的口径作废。账号入口需要体现完整用户资料采集和基础安全校验，不能只有用户名和密码两个输入框。

当前登录页截图暴露出的问题需要一并修正：

- 页面过于空旷，主体内容像临时居中占位。
- 左侧品牌区信息太少，logo 临时感强。
- 表单只是基础组件堆叠，缺少可交付产品入口质感。
- 测试账号不应作为正式界面内容展示。

## 后端接口

后端上下文路径为 `/campusops`，前端 `request` 已配置 `baseURL: '/campusops'`。

### 图形验证码

```text
GET /campusops/auth/captcha
```

前端调用：

```text
GET /auth/captcha
```

响应 `data`：

```json
{
  "captchaId": "uuid",
  "imageBase64": "data:image/png;base64,...",
  "expiresIn": 120
}
```

登录和注册都必须使用图形验证码。验证码图片应支持点击刷新。

### 注册

```text
POST /campusops/auth/register
```

前端调用：

```text
POST /auth/register
```

请求：

```json
{
  "username": "student002",
  "password": "123456",
  "confirmPassword": "123456",
  "realName": "王同学",
  "email": "student002@campusops.local",
  "phone": "13800000007",
  "department": "计算机学院",
  "captchaId": "uuid",
  "captchaCode": "A7K2"
}
```

注册成功后不自动登录，前端提示注册成功，并切换到登录表单。

### 登录

```text
POST /campusops/auth/login
```

前端调用：

```text
POST /auth/login
```

请求：

```json
{
  "account": "student002",
  "password": "123456",
  "captchaId": "uuid",
  "captchaCode": "A7K2"
}
```

`account` 支持用户名、手机号或邮箱。

### 当前用户

```text
GET /campusops/auth/me
```

前端调用：

```text
GET /auth/me
```

请求头：

```text
Authorization: Bearer <accessToken>
```

## 实现范围

### 1. 账号入口页

登录和注册采用独立页面，通过链接跳转：

- `LoginView.vue`（路径 `/login`）：登录表单，底部提供”还没有账号？创建账号”链接跳转 `/register`。
- `RegisterView.vue`（路径 `/register`）：注册表单，底部提供”已有账号？去登录”链接跳转 `/login`。
- 两个页面都必须有 loading 状态、清晰错误提示和基础校验。
- 不展示测试账号列表、快捷登录、调试说明。

登录字段：

- `account`：用户名 / 手机号 / 邮箱
- `password`
- `captchaCode`

登录请求体还需包含 `captchaId`，值取自验证码接口响应。

注册字段：

- `username`
- `phone`
- `email`
- `password`
- `confirmPassword`
- `realName`
- `department`（可选）
- `captchaCode`

注册请求体还需包含 `captchaId`，值取自验证码接口响应。

注册校验：

- 用户名、密码、确认密码、姓名、手机号、邮箱必填。
- 院系/部门可选。
- 两次密码必须一致。
- 手机号需要做格式校验。
- 邮箱必须符合邮箱格式。
- 验证码必填。

验证码交互：

- 进入登录页或注册页时自动获取验证码。
- 登录页和注册页之间跳转时会重新获取验证码。
- 点击验证码图片可以刷新验证码。
- 登录或注册失败后刷新验证码，避免用户重复提交已失效验证码。

### 2. API 封装

修改 `campusops-front/src/api/auth.js`：

- 保留 `login(data)`。
- 保留 `getCurrentUser()`。
- 新增 `register(data)`。
- 新增 `getCaptcha()`。
- 移除或停止使用不存在的后端 `logout()` 调用。

### 3. 用户状态管理

修改 `campusops-front/src/stores/user.js`：

- 保留 `token` 和 `userInfo`。
- 新增或完善 `loginAction(credentials)`、`registerAction(payload)`、`fetchCurrentUser()`、`logoutAction()`、`clearSession()`。
- 登录提交给后端的字段使用 `account`，不是旧的 `username`。
- Token 继续存储在 `localStorage.campusops_token`。
- `logoutAction` 只做前端本地退出，不调用后端 logout 接口。

### 4. 请求和路由

修改 `campusops-front/src/api/request.js`：

- 请求拦截器设置 `Authorization: Bearer <token>`。
- 响应拦截器统一返回后端响应体。
- `UNAUTHORIZED`、HTTP 401 或 Token 失效时，清理本地登录态并跳转 `/login`。
- 错误消息优先使用后端返回的 `message`。

修改 `campusops-front/src/router/index.js`：

- `/login` 是公开页面。
- 除 `/login` 外的工作台页面都需要登录。
- 未登录访问受保护页面时跳转 `/login`，并保留 `redirect` 查询参数。
- 已登录但 `userInfo` 为空时，调用 `fetchCurrentUser()` 恢复当前用户。
- 已登录访问 `/login` 时跳转 `/dashboard`。

### 5. 主界面用户信息

修改 `campusops-front/src/layouts/BasicLayout.vue`：

- 顶部右侧显示当前用户真实姓名；没有真实姓名时显示用户名。
- 可显示部门和角色，保持克制。
- 退出登录时清理本地状态并跳转 `/login`。
- 修复中文乱码和异常标签。

## Logo 与视觉资产

项目需要正式一点的品牌识别，不能只用蓝色方块字母 `C`。

本任务至少完成一种：

- 使用纯前端 SVG/CSS 做一个简洁 logo，放在 `public/favicon.svg` 或可复用组件中。
- 或使用生成图片作为登录页左侧视觉，图片必须提交到 `campusops-front/public/**`，并保证构建可用。

Logo 方向建议：

- 校园服务、工单流转、连接节点、盾牌/勾选、建筑轮廓等抽象元素。
- 不要卡通化，不要过度科技感，不要使用复杂 3D。
- 保持在侧边栏、登录页和 favicon 上都能识别。

如果使用 AI 生成图片，只建议生成一张克制的校园 IT 服务插图或背景图，不要用来替代核心 UI 设计。图片应避免真实学校商标、真实人物肖像和不可控文字。

## 页面设计要求

CampusOps 是校园 IT 服务工作台，不是营销落地页。

推荐方向：

- 登录和注册为独立页面（`/login` 和 `/register`），通过底部链接互相跳转。
- 居中卡片布局：品牌区（logo + 主标题 + 副标题）水平排列在卡片正上方。
- 卡片宽度 440px，注册页标题"创建账号"显示在卡片内部顶部。
- 背景图完全展示，卡片自身白色背景保证表单可读性。
- 底部显示版权信息。
- 移动端卡片撑满屏幕宽度，品牌区保持在卡片上方。

视觉禁忌：

- 不做营销落地页。
- 不用夸张渐变背景、大面积紫蓝光效、装饰性圆球、玻璃拟态卡片。
- 不堆砌“AI 赋能、智能、高效”等泛化口号。
- 不使用巨大 hero 标题挤占表单空间。
- 不把测试账号、调试说明、快捷登录暴露在正式 UI。
- 卡片圆角不超过 8px。

文案建议：

- 主标题：`CampusOps`
- 副标题：`校园 IT 服务工单平台`
- 能力摘要：
  - `提交问题，跟踪处理进度`
  - `统一管理校园 IT 服务请求`
  - `面向师生和运维团队的协作入口`

## 明确不要做

- 不做忘记密码流程。
- 不做短信验证码。
- 不做邮箱验证码。
- 不做 SSO。
- 不做工单列表、详情、创建页面的接口接入。
- 不修改后端。
- 不新增依赖。
- 不展示测试账号。

## 验收标准

必须全部满足：

- `npm run build` 通过。
- 页面进入时调用 `GET /campusops/auth/captcha` 获取验证码。
- 注册表单调用 `POST /campusops/auth/register`。
- 注册请求包含 `captchaId` 和 `captchaCode`。
- 注册成功后提示用户并跳转到登录页。
- 登录表单调用 `POST /campusops/auth/login`。
- 登录请求使用 `account`、`password`、`captchaId`、`captchaCode`。
- 登录输入框文案体现支持”用户名 / 手机号 / 邮箱”。
- 注册表单手机号和邮箱为必填项，有格式校验。
- 登录成功后 Token 写入 `localStorage.campusops_token`。
- 后续请求带 `Authorization: Bearer <token>`。
- 登录成功后调用 `GET /campusops/auth/me` 并保存当前用户。
- 未登录访问 `/dashboard`、`/tickets`、`/knowledge`、`/ai-analysis` 会跳转 `/login`。
- 已登录刷新页面后能恢复当前用户信息。
- 退出登录会清理 Token 和用户信息，并跳转 `/login`。
- 登录/注册入口页和基础布局没有中文乱码。
- 登录/注册入口页没有测试账号展示。
- 页面有可识别 logo 或品牌视觉资产。
- 1366px 桌面和 390px 移动宽度下，界面不空旷、不溢出、不遮挡。
