# 006-用户注册接口

## 任务目标

在 `campusops-server/` 中补充普通用户自助注册接口，形成真实的注册、登录、进入系统闭环。

本任务补齐账号体系的基础能力：验证码、普通用户注册、账号密码登录增强。

本任务不做管理员创建用户、不做角色管理、不做邮箱/短信验证码、不做找回密码。验证码采用后端生成的一次性图形验证码，不依赖第三方服务。

## 接口设计

### 获取图形验证码

```text
GET /campusops/auth/captcha
```

Controller 实际映射：

```text
GET /auth/captcha
```

响应 `data`：

```json
{
  "captchaId": "uuid",
  "imageBase64": "data:image/png;base64,...",
  "expiresIn": 120
}
```

规则：

- 验证码为 4 位随机字符或简单算术图形验证码。
- 验证码有效期 120 秒。
- 验证码校验成功或失败次数达到上限后应失效。
- 不引入 Redis；当前阶段可使用 JVM 内存存储验证码，后续多实例部署时再替换为 Redis 或集中式存储。

### 登录增强

既有登录接口继续使用：

```text
POST /campusops/auth/login
```

请求体调整为：

```json
{
  "account": "student001",
  "password": "123456",
  "captchaId": "uuid",
  "captchaCode": "A7K2"
}
```

兼容规则：

- `account` 支持用户名、手机号或邮箱。
- 为降低改动风险，可以临时兼容旧字段 `username`，但前端新实现必须使用 `account`。
- 密码仍使用现有 `PasswordEncoder.matches` 校验。
- 登录前必须先校验验证码；验证码错误不进入密码校验。

### 注册普通用户

```text
POST /campusops/auth/register
```

Controller 实际映射：

```text
POST /auth/register
```

请求体：

```json
{
  "username": "student002",
  "phone": "13800000007",
  "email": "student002@campusops.local",
  "password": "123456",
  "confirmPassword": "123456",
  "realName": "王同学",
  "department": "计算机学院",
  "captchaId": "uuid",
  "captchaCode": "A7K2"
}
```

字段规则：

- `username`：必填，长度 4-64，只允许字母、数字、下划线。
- `phone`：必填，长度不超过 32，且必须符合手机号格式。
- `email`：必填，必须符合邮箱格式。
- `password`：必填，长度 6-32。
- `confirmPassword`：必填，必须与 `password` 一致。
- `realName`：必填，长度 1-64。
- `department`：可选，长度不超过 128。
- `captchaId`：必填。
- `captchaCode`：必填。

响应 `data`：

```json
{
  "id": 7,
  "username": "student002",
  "realName": "王同学"
}
```

## 业务规则

- 注册用户默认 `status = enabled`。
- 注册用户默认绑定 `normal_user` 角色。
- 密码必须使用当前项目 `PasswordEncoder` 编码后写入 `sys_user.password_hash`。
- 用户名必须唯一，重复时返回统一业务错误。
- 邮箱不能与未删除用户重复。
- 手机号不能与未删除用户重复。
- 角色 `normal_user` 必须存在；不存在时返回系统配置错误或业务错误，不静默创建角色。
- 注册成功后不自动登录，由前端引导用户登录，避免注册接口承担 Token 生成逻辑。
- 注册前必须校验验证码；验证码错误不写入用户数据。

## 日志要求

- `AuthServiceImpl`、`CaptchaServiceImpl` 等涉及业务流程的实现类使用 `@Slf4j`。
- 验证码生成成功可记录 `log.info`，字段包含 `captchaId` 和过期时间，不记录验证码答案。
- 注册成功记录 `log.info`，字段包含 `userId`、`username`、`phone`、`email`，不记录密码。
- 登录成功记录 `log.info`，字段包含 `userId`、`username` 或账号类型，不记录 Token。
- 验证码错误、账号不存在、密码错误、用户名/邮箱/手机号重复等可预期失败记录 `log.warn`，字段包含错误码和必要定位字段。
- 不得记录明文密码、验证码答案、JWT Token、完整请求体。

## 测试要求

至少覆盖：

- 注册成功后 `sys_user` 新增用户，密码不是明文。
- 注册成功后绑定 `normal_user` 角色。
- 用户名重复时注册失败。
- 邮箱重复时注册失败。
- 手机号重复时注册失败。
- 两次密码不一致时注册失败。
- 验证码错误时注册失败，且不写入用户。
- 登录支持用户名、邮箱、手机号三种账号标识。
- 登录验证码错误时失败，且不进入密码校验。
- 注册后可使用新账号登录，并可访问 `/auth/me`。
