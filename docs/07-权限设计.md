# 权限设计

## 1. 设计目标

权限设计用于保证不同角色只能访问和操作自己被授权的数据，尤其要防止 AI 模块绕过权限边界。

系统第一版采用 RBAC 为主、数据权限规则为辅的权限模型。

## 2. 权限模型

### 2.1 认证

第一版建议使用账号密码登录和 JWT Token 鉴权。

认证流程：

1. 用户提交用户名和密码。
2. 后端校验用户状态和密码。
3. 校验通过后生成访问令牌。
4. 前端请求接口时携带令牌。
5. 后端解析令牌并加载用户角色和权限。

### 2.2 授权

授权分为两层：

- 接口权限：判断用户是否可以访问某类接口。
- 数据权限：判断用户是否可以访问某条具体业务数据。

接口权限适合通过 Spring Security 注解、配置或过滤器实现。

数据权限应在 Service 层显式校验，避免只依赖前端隐藏按钮。

## 3. 角色权限矩阵

| 功能 | 普通用户 | IT 处理人 | 服务管理员 | 系统管理员 |
|---|---|---|---|---|
| 提交工单 | 是 | 是 | 是 | 可选 |
| 查看自己的工单 | 是 | 是 | 是 | 可选 |
| 查看分配给自己的工单 | 否 | 是 | 是 | 可选 |
| 查看全部工单 | 否 | 否 | 是 | 可选 |
| 分派工单 | 否 | 否 | 是 | 否 |
| 转派工单 | 否 | 否 | 是 | 否 |
| 处理工单 | 否 | 是 | 是 | 否 |
| 标记已解决 | 否 | 是 | 是 | 否 |
| 用户确认关闭 | 是，仅自己的工单 | 否 | 否 | 否 |
| 管理服务分类 | 否 | 否 | 是 | 否 |
| 管理知识库文章 | 查看已发布 | 创建草稿 | 审核发布 | 可选 |
| 查看统计报表 | 否 | 仅个人 | 是 | 可选 |
| 用户管理 | 否 | 否 | 否 | 是 |
| 角色权限管理 | 否 | 否 | 否 | 是 |
| 查看 AI 调用日志 | 否 | 仅相关工单 | 是 | 是 |

## 4. 工单数据权限

### 4.1 普通用户

普通用户只能访问自己提交的工单。

允许：

- 查看自己的工单。
- 修改未处理前的标题和描述。
- 添加评论和附件。
- 确认关闭自己的工单。
- 取消未进入处理中的工单。

### 4.2 IT 处理人

IT 处理人可以访问：

- 分配给自己的工单。
- 服务管理员授权的可接单工单。

允许：

- 接单。
- 添加处理记录。
- 回复用户。
- 标记已解决。
- 创建知识库草稿。

### 4.3 服务管理员

服务管理员可以访问全部工单。

允许：

- 分派工单。
- 转派工单。
- 调整分类。
- 调整优先级。
- 驳回无效工单。
- 查看统计报表。

### 4.4 系统管理员

系统管理员主要管理系统配置，不默认拥有全部业务操作权。

如果系统管理员需要参与工单处理，应额外授予服务管理员或 IT 处理人角色。

## 5. 状态流转权限

| 状态变化 | 操作 | 允许角色 |
|---|---|---|
| 无 -> 待分派 | 创建工单 | 普通用户、IT 处理人、服务管理员 |
| 待分派 -> 待处理 | 分派工单 | 服务管理员 |
| 待处理 -> 处理中 | 接单 | 当前处理人 |
| 处理中 -> 待用户确认 | 标记已解决 | 当前处理人、服务管理员 |
| 待用户确认 -> 已关闭 | 确认关闭 | 工单提交人 |
| 待分派 -> 已驳回 | 驳回工单 | 服务管理员 |
| 待分派/待处理 -> 已取消 | 取消工单 | 工单提交人 |

所有状态变化必须记录 `ticket_record` 和 `operation_log`。

## 6. AI 数据权限

AI 模块必须遵循以下规则：

- AI 分析当前工单前，必须校验当前用户是否有权访问该工单。
- AI 推荐相似工单时，推荐结果必须经过数据权限过滤。
- AI 推荐知识库文章时，普通用户只能看到已发布文章。
- AI 生成知识库草稿时，不得自动发布。
- AI 不得直接调用绕过权限校验的数据库查询。
- AI 调用日志不应记录完整敏感内容。

## 7. 实现建议

- Controller 层负责认证用户获取和基础参数校验。
- Service 层负责业务权限和数据权限校验。
- Repository / Mapper 层不直接暴露无权限过滤的大范围查询给 Controller。
- 高频权限判断可以封装为独立 `PermissionService`。
- 工单访问判断可以封装为 `TicketAccessService`。

建议提供方法：

```text
canViewTicket(userId, ticketId)
canAssignTicket(userId, ticketId)
canProcessTicket(userId, ticketId)
canCloseTicket(userId, ticketId)
canViewAiAnalysis(userId, ticketId)
```

## 8. 权限测试要求

必须覆盖：

- 普通用户不能查看他人工单。
- 普通用户不能分派工单。
- IT 处理人不能处理未分配给自己的工单。
- 服务管理员可以分派和转派工单。
- 系统管理员不应默认绕过业务权限。
- AI 推荐结果不能包含当前用户无权访问的工单。
