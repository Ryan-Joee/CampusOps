# AI Agent 协作规范

## 1. 文档目的

本文档用于约束 AI Agent 参与本项目开发时的行为，帮助项目在使用 AI 辅助开发的情况下保持架构一致、边界清晰和质量可控。

## 2. 开发前必须阅读

AI Agent 在执行较大开发任务前，应优先阅读以下文档：

- `docs/00-项目总览.md`
- `docs/01-需求说明.md`
- `docs/02-用户角色与权限边界.md`
- `docs/03-业务流程.md`
- `docs/04-系统架构设计.md`
- `docs/08-AI模块设计.md`
- `docs/14-后端分层与包结构规范.md`

涉及数据库、接口、测试、部署时，应阅读对应专项文档。

## 3. 基本原则

- 不在不了解上下文的情况下直接大规模修改代码。
- 优先遵循项目已有架构和命名规范。
- 后端代码必须遵循 `docs/14-后端分层与包结构规范.md`。
- 不绕过权限设计实现业务功能。
- 不把 AI 能力写散在 Controller 中。
- 不让 AI 模块直接修改关键业务状态。
- 不提交 `.agents/`、`.codex/`、`.claude/` 等本地 Agent 工具目录。
- 重要设计变更应补充 ADR。
- 功能开发应同步考虑测试和文档更新。

## 4. 任务拆分原则

AI Agent 执行任务时，应尽量拆分为小步：

1. 阅读相关文档和代码。
2. 明确影响范围。
3. 给出实现方案。
4. 修改代码。
5. 补充或更新测试。
6. 运行验证。
7. 更新相关文档。

## 5. 权限相关约束

权限逻辑属于高风险区域。

修改以下内容时必须谨慎：

- 登录认证。
- 角色权限。
- 数据权限。
- 工单可见性规则。
- AI 可访问数据范围。

任何 AI 推荐、摘要、检索和问答能力，都必须先经过权限过滤。

## 6. AI 模块开发约束

AI 模块应通过独立 Service 或 Adapter 进行封装。

不允许：

- 在 Controller 中直接拼接 prompt 并调用模型。
- 在业务代码中散落模型 API Key。
- 将模型输出直接作为正式业务状态。
- 忽略 AI 调用失败情况。

必须考虑：

- 输入校验。
- 输出格式校验。
- 调用日志。
- 失败降级。
- 用户权限。
- 人工确认。

## 7. 测试要求

新增功能应优先补充以下测试：

- Service 层业务测试。
- Controller 接口测试。
- 权限访问测试。
- 工单状态流转测试。
- AI 输出格式测试。

如果暂时无法补测试，应在任务结果中说明原因和风险。

## 8. 文档更新要求

当功能、接口、数据库、权限或 AI 能力发生变化时，应同步更新相关文档。

常见更新位置：

- 需求变化：`docs/01-需求说明.md`
- 角色权限变化：`docs/02-用户角色与权限边界.md`
- 流程变化：`docs/03-业务流程.md`
- 架构变化：`docs/04-系统架构设计.md`
- AI 能力变化：`docs/08-AI模块设计.md`
- 重要决策：`docs/adr/`
