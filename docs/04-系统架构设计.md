# 系统架构设计

## 1. 架构目标

系统架构应满足以下目标：

- 支撑校园 IT 工单处理的核心业务闭环。
- 保持传统业务模块和 AI 能力模块的清晰边界。
- 支持权限控制、日志审计、测试和后续扩展。
- 支持未来演进到 RAG、工具调用、Agent 和 MCP。

## 2. 后端分层

后端采用常见 Spring Boot 分层结构：

- Controller：负责 HTTP 接口和请求参数校验。
- Service：负责业务逻辑和事务控制。
- Repository / Mapper：负责数据库访问。
- Domain / Entity：负责领域对象和数据实体。
- DTO / VO：负责接口输入输出模型。
- Infrastructure：负责外部服务、缓存、文件、AI 模型调用等基础设施能力。

## 3. 核心模块

### 3.1 用户权限模块

负责用户、角色、权限和登录认证。

主要能力：

- 用户管理。
- 角色管理。
- 登录认证。
- 接口权限控制。
- 数据权限控制。

### 3.2 工单模块

负责工单主业务流程。

主要能力：

- 工单创建。
- 工单查询。
- 工单分派。
- 工单状态流转。
- 处理记录。
- 评论与附件。
- 超时判断。

### 3.3 知识库模块

负责 IT 服务经验的沉淀和查询。

主要能力：

- 知识库文章管理。
- 分类和标签管理。
- 文章发布与下线。
- 工单关联知识库。
- 从工单生成知识库草稿。

### 3.4 AI 辅助模块

负责封装 AI 能力，不直接承担核心业务状态变更。

主要能力：

- 工单分类建议。
- 工单摘要。
- 关键词提取。
- 相似工单推荐。
- 知识库推荐。
- AI 调用日志。
- 后续 RAG 和 Agent 能力扩展。

### 3.5 统计分析模块

负责服务运营数据统计。

主要能力：

- 工单状态统计。
- 分类占比统计。
- 处理时长统计。
- 超时工单统计。
- 处理人工作量统计。

### 3.6 日志审计模块

负责记录关键操作和系统行为。

主要能力：

- 工单操作日志。
- 用户登录日志。
- 权限变更日志。
- AI 调用日志。

### 3.7 通知模块

负责工单状态变化提醒。

第一版可以先实现站内通知，后续再接入邮件、短信、企业微信或钉钉。

## 4. AI 模块边界

AI 模块只能生成建议、摘要、草稿和推荐结果。

AI 模块不能直接绕过业务 Service 修改正式业务状态。

需要修改业务状态时，必须通过后端业务接口，并由具备权限的人类用户确认。

## 5. 数据流概览

普通工单处理数据流：

```text
用户提交工单
-> 工单模块保存数据
-> AI 模块生成辅助分析
-> 管理员分派工单
-> 处理人处理工单
-> 用户确认关闭
-> 日志审计模块记录操作
-> 知识库模块沉淀经验
```

AI 推荐数据流：

```text
工单内容
-> 权限过滤
-> AI 辅助模块
-> 分类/摘要/推荐结果
-> 保存 AI 分析记录
-> 用户查看并人工确认
```

## 6. 初步技术选型

- 后端框架：Spring Boot 3.5.15
- Java 版本：Java 17
- 权限认证：Spring Security
- 数据库：MySQL
- ORM：MyBatis Plus
- 缓存：Redis
- 接口文档：OpenAPI / Swagger
- AI 集成：Spring AI 或 LangChain4j
- 测试：JUnit 5
- 部署：Docker Compose

Redis、OpenAPI、AI 集成和 Docker Compose 根据实现阶段逐步引入，不在第一阶段提前添加依赖。具体后端技术选型见 `docs/adr/0002-后端技术选型.md`。
